How I Design Data Platforms

My day-to-day work centers on a Source-to-Core Data Product pattern: ingest data from relational source systems, run PySpark transformations with schema enforcement and deduplication, and publish reliable Core Data Products to downstream consumers. 

graph LR
    A[Source Systems] --> B[PySpark ETL]
    B --> C[Core Data Product]
    C --> D[Data Consumers]
    E[Kafka Events] --> B
    F[S3A Object Storage] --> B
    G[GitLab CI/CD] --> H[Docker Spark Jobs]
    H --> B
    I[HashiCorp Vault] --> B
  • Configuration-driven: JSON job definitions make pipelines easy to extend, review, and deploy across environments.
  • Exactly-once processing: checkpointing, idempotent writes, and watermark-based deduplication keep data reliable.
  • Secure by default: secrets, certificates, and credentials are fetched from HashiCorp Vault at runtime.
  • Multi-environment: the same Docker image moves through dev, sit, perf, and prod via GitLab CI/CD.
  • Observable: every job is wrapped with logging, retry logic, and status tracking.
